SDD Подорож: Автоматизація життєвого циклу публікацій

Сьогодні ми завершили впровадження критичної фічі для vadpoint.com — автоматизованого контролю стану публікацій. Це була цікава задача, яка поєднала в собі статику Astro та динаміку GitHub Actions.

Архітектурні рішення

  1. Гейт для публікації: Тепер сайт будується лише з матеріалів, що мають статус ready_for_publish або published. Це дозволяє автору спокійно працювати над чернетками (draft), не боячись випадкового оприлюднення.
  2. Зворотний зв’язок (Feedback Loop): Найскладнішою частиною було забезпечення синхронізації стану. Ми налаштували GitHub Actions так, щоб після успішного деплою пайплайн сам повертався в репозиторій і оновлював статус файлів з ready_for_publish на published.
  3. Обробка помилок: Якщо білд або деплой падає, система автоматично маркує файли як error_in_build або error_in_deploy. Це забезпечує прозорість для автора прямо в Obsidian.

Технічні деталі

  • Astro Filter: Використано функцію фільтрації в getCollection для глобального контролю контенту.
  • Node.js Script: Написано кастомний скрипт update-state.mjs на базі js-yaml для безпечного маніпулювання front-matter без пошкодження структури файлу.
  • GitHub Actions: Workflow був розширений кроками grep для пошуку активних файлів та використанням stefanzweifel/git-auto-commit-action для автоматичного пушу змін назад у main з тегом [skip ci].

Виклики

Основним викликом було налаштування імпортів для скрипта оновлення стану, оскільки залежність js-yaml знаходилась всередині папки engine/, а скрипт мав працювати з кореня репозиторію. Рішенням стало використання абсолютних шляхів до модулів всередині скрипта.

Тепер процес публікації виглядає так: написав -> поставив статус -> пушнув -> отримав підтвердження прямо у файлі. Справжній Spec-Driven Development у дії!
