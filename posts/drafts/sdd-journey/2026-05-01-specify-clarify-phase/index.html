SDD Подорож: Від сирих ідей до загартованої специфікації

Ми щойно завершили критично важливий етап у життєвому циклі vadpoint.com — перехід від “хайпових” концепцій до структурованої, валідованої специфікації архітектури (001-architecture-pipeline). Це саме той момент, де Spec-Driven Development (SDD) показує свою справжню силу, відфільтровуючи шум і залишаючи чисту логіку.

Фаза 1: Специфікація (Specify)

Ми взяли сирі нотатки про Astro, Obsidian та GitHub Actions і перетворили їх на контракт. Основні архітектурні рішення, які були зафіксовані:

  • SSG: Використання Astro для генерації статичного сайту з Zero JavaScript за замовчуванням.
  • CMS: Obsidian як Single Source of Truth. Весь контент — це чистий Markdown.
  • Sync: Автоматизація через Obsidian Git (15-хвилинні цикли) та GitHub Actions для CI/CD.

Це не просто вибір інструментів, це дотримання нашої Конституції: KISS Above All. Жодних баз даних, жодних складних серверних рантаймів на MVP.

Фаза 2: Уточнення (Clarify)

Навіть найкращі специфікації мають “сліпі зони”. Через ітеративний процес уточнення ми закрили два фундаментальні питання:

  1. Routing: Ми обрали Root-based routing для української мови (/about) та префіксацію для англійської (/en/about). Це найкраще рішення для SEO та чистоти URL.
  2. Data Model: Вирішено використовувати Unified pages collection. Це максимально спрощує першу ітерацію, дозволяючи Astro оперувати єдиною схемою Zod для всіх типів контенту на старті.

Чому це важливо?

Як інженер з 30-річним досвідом, я знаю: ціна помилки на етапі планування в 100 разів нижча, ніж на етапі коду. Ми делегували ШІ не просто написання коду, а архітектурне міркування. Тепер наш “контракт” готовий до виконання.
