SDD Journey: Перехід на Bun у CI та вирішення конфліктів залежностей

Сьогодні ми зіткнулися з каскадом помилок у GitHub Actions, які заблокували наш деплой. Це чудовий приклад того, як розбіжність між локальним середовищем та CI може призвести до проблем.

Аналіз помилок

  1. npm ci vs bun.lock: Команда npm ci вимагає наявності package-lock.json. Оскільки наш проект використовує Bun (і має лише bun.lock), npm не знав, як виконати “чисту інсталяцію”.
  2. Конфлікт Peer Dependencies: При спробі виконати звичайний npm install, виникла помилка ERESOLVE. Пакет @astrojs/check вимагав typescript: ^5.0.0, тоді як у нашому проекті вже встановлено typescript: ^6.0.3 (що є дуже свіжою версією для 2026 року). npm за замовчуванням блокує такі інсталяції.

Чому Bun кращий для нашого CI?

Локально ви вже використовуєте Bun, що підтверджується наявністю bun.lock та Bun-скриптів у package.json. Bun набагато лояльніше ставиться до розбіжностей у версіях peer-залежностей і, що важливо, він працює значно швидше за npm.

Що було зроблено:

  • deploy.yml: Ми замінили actions/setup-node на oven-sh/setup-bun.
  • Install & Build: Тепер замість npm використовуються команди bun install та bun run build.

Це рішення повністю синхронізує ваш локальний процес розробки з процесом у хмарі, усуваючи будь-які несподіванки при деплої.
