SDD Journey: Культура Git та чи потрібні нам Pull Requests?

Працюючи за методологією Spec-Driven Development, ми часто ставимо питання: чи не є Pull Requests (PR) зайвою бюрократією для соло-розробника? Давайте розберемося, як це працює в нашому AI-Native контексті.

Чи потрібен PR?

Для проекту vadpoint.com відповідь — так, але як опція для валідації.

Хоча ви — єдиний автор, використання PR дає кілька критичних переваг:

  1. Автоматична перевірка: GitHub Actions запустить білд Astro ще до того, як зміни потраплять у main. Якщо ми десь помилилися у frontmatter, ми дізнаємося про це до того, як “зламаємо” публічний сайт.
  2. Чиста історія: Всі дрібні комміти ітерацій ШІ ховаються всередині одного PR, а в main потрапляє один чистий Squash merge з посиланням на номер специфікації.
  3. SDD Compliance: Це дозволяє закрити цикл: “Специфікація -> Гілка -> Реалізація -> Review -> Merge”.

Як правильно робити PR у нашому проекті?

Ось рекомендований алгоритм дій після того, як ШІ завершив роботу в гілці:

  1. Комміт змін: (Те, що ми робимо зараз).
  2. Push гілки: 
    git push origin <branch-name>
  3. Створення PR: На сторінці репозиторію в GitHub з’явиться пропозиція “Compare & pull request”. У назві PR обов’язково вказуйте номер фічі, наприклад: feat: cleanup pipeline [007].
  4. Очікування CI: Дочекайтеся зеленої галочки від GitHub Actions (Build Astro).
  5. Merge: Використовуйте Squash and merge для збереження чистоти історії.
  6. Видалення гілки: Після мерджу гілка більше не потрібна.

Висновок

Для дрібних правок контенту можна мерджити відразу. Але для архітектурних змін (як сьогоднішнє очищення пайплайну) PR — це ваш останній “запобіжник”, який гарантує, що сайт не впаде після деплою.
